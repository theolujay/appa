# Appa Ansible

This directory contains Ansible playbooks and roles for host-level Appa server
hardening. It is intended to be called by the future Appa CLI, but can also be
run directly against a generated or local test inventory.

## Playbooks

- `playbooks/security-hardening.yml` applies kernel, account, SSH, firewall, and
  audit hardening.
- `playbooks/compliance-scan.yml` performs read-only checks for the same control
  areas and reports drift.

## Local Test Run

The Vagrant harness in `dev/` boots an Ubuntu VM and applies the hardening
playbook:

```bash
make vagrant/up          # from repo root (auto-adds venv to PATH)
# or manually:
cd deploy/ansible/dev
source ../.venv/bin/activate && vagrant up
```

For direct runs from `deploy/ansible`:

```bash
ansible-playbook -i dev/inventory.ini playbooks/security-hardening.yml
ansible-playbook -i dev/inventory.ini playbooks/compliance-scan.yml
```

### Known Issues

**VirtualBox Guest Additions version mismatch**

The bento box ships with Guest Additions 7.2.4, while the host may run a newer
VirtualBox (e.g. 7.2.8). The `vagrant-vbguest` plugin auto-matches them, but the
box's stale apt cache can cause 404 errors when installing build dependencies.

**Workaround:** If Vagrant times out or SSH resets repeatedly:

1. Install the plugin: `vagrant plugin install vagrant-vbguest`
2. Destroy and rebuild: `make vagrant/destroy && make vagrant/up`
3. If 404 errors appear, the VM may need a full `apt-get update && apt-get upgrade`
   first. Temporarily uncomment `v.cpus = 2` in `dev/Vagrantfile` and retry.

## Important Variables

- `ssh_port`: SSH daemon port, default `22`.
- `ssh_allowed_users`: users allowed to log in over SSH, default `['deploy']`.
- `allow_root_ssh_keys`: allow root key-based SSH login, default `false`.
- `allow_password_auth`: allow SSH password authentication, default `false`.
- `firewall_backend`: `ufw`, `firewalld`, or `iptables`; defaults to `ufw` on
  Debian/Ubuntu hosts and `firewalld` elsewhere.
- `ufw_rules`: Ubuntu/Debian allow rules for SSH, HTTP, and HTTPS.
- `firewall_rules`: firewalld public and internal allow rules.
- `sudo_rules`: optional additional sudo rules.

Production inventory and secrets should be generated by the CLI or supplied by
the operator. Do not commit real host credentials, provider tokens, SMTP
passwords, or backup keys.
