# Secrets Management

This document describes how Appa manages secrets across the platform stack and user
application deployments. It covers the current state, the phased migration to Docker
Swarm Secrets, and the eventual plan for user-deployed application secrets.

## Current State

### Platform secrets

The Ansible deploy playbook generates a shared `.env` file rendered from `env.j2`
and mounted into every stack service via `env_file`. The file contains:

| Variable | Source | Sensitivity |
|---|---|---|
| `APPA_DB_DSN` | Derived from generated `pg_password` | HIGH — full DB access |
| `POSTGRES_PASSWORD` | Generated by Ansible, stored on disk | HIGH — database superuser |
| `SMTP_PASSWORD` | User-provided | HIGH — mail relay access |
| `CLOUDFLARE_TOKEN` | User-provided | HIGH — DNS/TLS control |
| Everything else | Config defaults or user-provided | LOW — tuning, routing |

There is **no Docker Secrets usage**. Every secret lives either in:
- The shared `.env` file on disk (readable by any process on the host with file access)
- The `project_envs` table in PostgreSQL (plaintext `TEXT` column)
- The `deployments.env_vars` column in PostgreSQL (plaintext `TEXT`, newline-delimited `KEY=VALUE`)

Ansible applies `no_log: true` when rendering `.env`, so secrets are masked in
Ansible output, but at rest they are plaintext.

### User app env vars

Env vars flow through the system as follows:

```
CLI/API request
  → project_envs table (project-level, plaintext)
  → deployments.env_vars (deployment-level, plaintext, merged with project vars)
  → pipeline builder: --env KEY=VAL to railpack prepare
  → pipeline runner: container.Config.Env = [KEY=VAL]
  → container runtime: plain Docker environment variables
```

There is **no distinction between plain env vars and secrets** at any layer.

## Terminology

| Term | Meaning |
|---|---|
| **Platform secrets** | Values the platform needs to operate: DB password, SMTP credentials, Cloudflare token. Managed by Ansible in the stack deploy. |
| **User app secrets** | Values a deployed application needs at runtime: API keys, database passwords, signing secrets. Managed through the `project env` CLI / API. |
| **Docker Secret** | A Swarm-native encrypted blob stored in the Raft log, mounted as a file at `/run/secrets/<name>` inside containers. Only visible to services that explicitly request it. |
| **Stack** | A Docker Compose / Stack file deployed via `docker stack deploy`. The Appa platform runs as two stacks: `appa_data` and `appa`. |

## Phase 1 — Platform Secrets → Docker Secrets

Move the three high-sensitivity platform values out of the shared `.env` file into
dedicated Docker Swarm secrets.

### New secrets

| Secret name | Content | Consumed by |
|---|---|---|
| `{{ stack_name }}_pg_password` | PostgreSQL password | `api`, `db` |
| `{{ stack_name }}_smtp_password` | SMTP password | `api` |
| `{{ stack_name }}_cloudflare_token` | Cloudflare API token | `api`, `caddy` |

### Files to change

#### `deploy/ansible/roles/appa_stack/tasks/main.yml`

After the PG password is generated (around line 45), create the secrets:

```yaml
- name: Create PostgreSQL password secret
  community.docker.docker_secret:
    name: "{{ appa_stack_name }}_pg_password"
    data: "{{ pg_password }}"
    state: present
  no_log: true

- name: Create SMTP password secret
  community.docker.docker_secret:
    name: "{{ appa_stack_name }}_smtp_password"
    data: "{{ smtp_password }}"
    state: present
  no_log: true

- name: Create Cloudflare token secret
  community.docker.docker_secret:
    name: "{{ appa_stack_name }}_cloudflare_token"
    data: "{{ cloudflare_token }}"
    state: present
  no_log: true
```

The secrets should be created **before** the stack templates are rendered and
deployed, so they exist in Swarm by the time the stack references them.

#### `stack.data.yml.j2`

Add a top-level `secrets:` and switch the `db` service from `env_file` to
explicit secrets:

```yaml
services:
  db:
    image: postgres:17-alpine
    volumes:
      - pg_data:/var/lib/postgresql/data
    secrets:
      - pg_password
    environment:
      POSTGRES_USER: appa
      POSTGRES_DB: appa
      POSTGRES_PASSWORD_FILE: /run/secrets/pg_password
    networks:
      - appa_net

secrets:
  pg_password:
    external: true
    name: "{{ appa_stack_name }}_pg_password"
```

Postgres supports `POSTGRES_PASSWORD_FILE` to read the password from a file
instead of an environment variable.

#### `stack.base.yml.j2`

Add `secrets:` at the top level and reference them in the `api` and `caddy`
services:

```yaml
services:
  api:
    image: ghcr.io/theolujay/appa:latest
    env_file: {{ appa_stack_dir }}/.env
    secrets:
      - pg_password
      - smtp_password
      - cloudflare_token
    ...

  caddy:
    image: caddy:2.11.2-alpine
    env_file: {{ appa_stack_dir }}/.env
    secrets:
      - cloudflare_token
    ...

secrets:
  pg_password:
    external: true
    name: "{{ appa_stack_name }}_pg_password"
  smtp_password:
    external: true
    name: "{{ appa_stack_name }}_smtp_password"
  cloudflare_token:
    external: true
    name: "{{ appa_stack_name }}_cloudflare_token"
```

#### `env.j2`

Remove the three secret values. Keep everything else:

```diff
- APPA_DB_DSN=postgres://appa:{{ pg_password }}@db:5432/appa?sslmode=disable
  POSTGRES_USER=appa
- POSTGRES_PASSWORD={{ pg_password }}
  POSTGRES_DB=appa
  PORT=8080
  ENV=production
  SMTP_HOST={{ smtp_host }}
  SMTP_PORT={{ smtp_port }}
- SMTP_USERNAME={{ smtp_username }}
- SMTP_PASSWORD={{ smtp_password }}
- SMTP_SENDER={{ smtp_sender }}
  CORS_TRUSTED_ORIGINS={{ cors_origins }}
  RAILPACK_VERSION={{ railpack_version }}
  LIMITER_RPS=2
  LIMITER_BURST=4
  LIMITER_ENABLED=true
  DB_MAX_OPEN_CONNS=25
  DB_MAX_IDLE_CONNS=25
  DB_MAX_IDLE_TIME=15m
- CLOUDFLARE_TOKEN={{ cloudflare_token }}
```

#### `entrypoint.sh.j2`

The API entrypoint script should now construct the DSN from the secret instead
of reading it from the environment:

```bash
#!/usr/bin/env bash
set -euo pipefail

PG_PASSWORD=$(cat /run/secrets/pg_password)
export APPA_DB_DSN="postgres://appa:${PG_PASSWORD}@db:5432/appa?sslmode=disable"

./appa-api
```

Alternatively, the Go binary itself can read the secret files at startup (see
below).

#### `cmd/api/main.go`

Add a helper to read secrets at startup and construct the DSN in Go instead of
shell:

```go
import (
    "os"
    "path/filepath"
    "strings"
)

func readSecret(name string) (string, error) {
    data, err := os.ReadFile(filepath.Join("/run/secrets", name))
    if err != nil {
        return "", err
    }
    return strings.TrimSpace(string(data)), nil
}
```

Then in `main()`, override the DSN from environment with one constructed from
secrets:

```go
if pgPassword, err := readSecret("pg_password"); err == nil {
    cfg.db.dsn = fmt.Sprintf(
        "postgres://appa:%s@db:5432/appa?sslmode=disable", pgPassword,
    )
}
if smtpPassword, err := readSecret("smtp_password"); err == nil {
    cfg.smtp.password = smtpPassword
}
```

This keeps secrets out of process environment variables entirely — they exist
only as files in `/run/secrets/`, read once at startup.

#### `Caddyfile.j2`

Switch from env var `{$CLOUDFLARE_TOKEN}` to Caddy's `file()` directive:

```diff
  tls {
-   dns cloudflare {$CLOUDFLARE_TOKEN}
+   dns cloudflare {file /run/secrets/cloudflare_token}
  }
```

Caddy's `file()` loader reads the content of the given path at TLS handshake
time, so the secret file can be rotated without restarting Caddy.

### Verification

After deployment, verify that:
1. `docker secret ls` shows the three secrets
2. Secrets are **not** present in `docker inspect <service>` env output
3. The API starts successfully and connects to the database
4. Caddy obtains TLS certificates via Cloudflare DNS-01
5. `docker exec <api_container> cat /run/secrets/pg_password` shows the value

## Phase 2 — User App Secrets (Swarm Services)

Phase 2 moves user application deployments from standalone Docker containers to
Docker Swarm services. This enables native Docker Secrets for per-deployment
sensitive env vars and unlocks rolling updates, health-check-aware zero-downtime
deploys, and integrated service management.

### Why Swarm services instead of standalone containers

| Feature | Standalone container | Swarm service |
|---|---|---|
| Docker Secrets | Not available | Native via `ContainerSpec.Secrets` |
| Rolling updates | Manual | Built-in `UpdateConfig` |
| Zero-downtime | Manual blue/green | `order: start-first` + health checks |
| Restart policy | `--restart` flag | `RestartPolicy` with conditions |
| Service discovery | Manual network alias | DNS round-robin via VIP |
| Scaling | Not supported | `ReplicatedService.R replicas` |
| Desired-state convergence | None | Swarm continuously reconciles |

### Pipeline changes

The core change is in `internal/pipeline/runner.go`. Instead of:

```go
// Standalone container
resp, err := cli.ContainerCreate(ctx, &container.Config{...}, ...)
cli.ContainerStart(ctx, resp.ID, ...)
```

Use Swarm service creation:

```go
import (
    "github.com/docker/docker/api/types/swarm"
    "github.com/docker/docker/api/types"
)

// Create Docker secrets for sensitive env vars
var secretRefs []*swarm.SecretReference
for key, value := range secretEnvVars {
    name := fmt.Sprintf("appa_deploy_%d_%s", d.ID, strings.ToLower(key))
    _, err := cli.SecretCreate(ctx, swarm.SecretSpec{
        Annotations: swarm.Annotations{Name: name},
        Data:        []byte(value),
    })
    secretRefs = append(secretRefs, &swarm.SecretReference{
        SecretName: name,
        File: &swarm.SecretReferenceFileTarget{
            Name: fmt.Sprintf("/run/secrets/%s", strings.ToLower(key)),
            UID:  "0", GID: "0", Mode: 0400,
        },
    })
}

// Build the service spec
one := uint64(1)
serviceSpec := swarm.ServiceSpec{
    TaskTemplate: swarm.TaskSpec{
        ContainerSpec: &swarm.ContainerSpec{
            Image:   *d.ImageTag,
            Env:     nonSecretEnv,        // only non-secret KEY=VAL strings
            Secrets: secretRefs,
            Healthcheck: &container.HealthConfig{
                Test:     []string{"CMD-SHELL", "wget -qO- http://localhost:3000/health || exit 1"},
                Interval: 10 * time.Second,
                Timeout:  5 * time.Second,
                Retries:  3,
            },
        },
        Networks: []swarm.NetworkAttachmentConfig{
            {Target: "appa_net"},
        },
        RestartPolicy: &swarm.RestartPolicy{
            Condition: swarm.RestartPolicyConditionAny,
        },
        Resources: &swarm.ResourceRequirements{
            Limits: &swarm.Resources{
                MemoryBytes: int64(256 * 1024 * 1024), // 256MB default
            },
        },
    },
    Mode: swarm.ServiceMode{
        Replicated: &swarm.ReplicatedService{Replicas: &one},
    },
    UpdateConfig: &swarm.UpdateConfig{
        Parallelism:     1,
        Delay:           10 * time.Second,
        FailureAction:   swarm.UpdateFailureActionRollback,
        Monitor:         30 * time.Second,
        Order:           swarm.UpdateOrderStartFirst,
    },
    Name: fmt.Sprintf("appa-deploy-%d", d.ID),
    EndpointSpec: &swarm.EndpointSpec{
        Mode: swarm.ResolutionModeVIP,
    },
}

// Create the service
resp, err := cli.ServiceCreate(ctx, serviceSpec, types.ServiceCreateOptions{})
serviceID := resp.ID
```

### Log streaming

Swarm services support log streaming via the same SDK — the service ID replaces
the container ID:

```go
logsResp, err := cli.ServiceLogs(ctx, serviceID, types.ContainerLogsOptions{
    ShowStdout: true,
    ShowStderr: true,
    Follow:     true,
    Tail:       "100",
})
```

### Stop / restart

Stop a deployment by removing the service and its secrets:

```go
// Scale to zero first (graceful)
cli.ServiceUpdate(ctx, serviceID, swarm.Version{Index: v}, swarm.ServiceSpec{
    Mode: swarm.ServiceMode{
        Replicated: &swarm.ReplicatedService{Replicas: &zero},
    },
}, types.ServiceUpdateOptions{})

// Or remove entirely
cli.ServiceRemove(ctx, serviceID)

// Clean up associated secrets
for _, secret := range deploymentSecrets {
    // List secrets with label or convention
    cli.SecretRemove(ctx, secretID)
}
```

Restart a deployment by removing the old service and creating a new one with the
same image tag (or a new build).

### Route registration

Service VIPs are stable as long as the service exists, so route registration
(Caddy upstream) remains similar. Use the service name (`appa-deploy-<ID>`)
instead of the container IP. Swarm DNS resolves the service name to the VIP.

### Build-time secrets

For `railpack prepare`, pass secrets via `--secret` instead of `--env` so they
don't leak into image layers:

```go
cmd.Args = append(cmd.Args, "--secret", fmt.Sprintf("id=%s,env=%s", key, key))
```

Railpack supports BuildKit's secret syntax. The secret value is available during
the build but is not written to any layer.

### Cleanup

When a deployment is fully removed (status transitions to `stopped` or
`failed`), clean up all associated Swarm resources:

1. Remove the service: `docker service rm appa-deploy-<ID>`
2. Remove the secrets: `docker secret rm appa_deploy_<ID>_<key>` for each key
3. Remove the image (optional): `docker rmi <image_tag>`

### Naming convention

| Resource | Pattern | Example |
|---|---|---|
| Service | `appa-deploy-<deployment_id>` | `appa-deploy-42` |
| Secret | `appa_deploy_<deployment_id>_<key_lower>` | `appa_deploy_42_db_password` |
| Image tag | Deployment record's `image_tag` | `appa:deploy-42-a1b2c3d` |

### Distinguishing secrets from plain vars

Since the original Phase 1 (model-level `is_secret` flag) was merged into Phase 2, the `project_envs` table will be
extended with an `is_secret` boolean:

```sql
ALTER TABLE project_envs ADD COLUMN is_secret BOOLEAN NOT NULL DEFAULT FALSE;
```

The CLI flag `project env set --secret` marks a variable as sensitive.
`MergeProjectEnvVars` returns two values: plain env vars and secret env vars.
The pipeline passes plain vars as `ContainerSpec.Env` and secret vars as
Docker Secrets.

The deployment request payload can also accept a `secret_keys` list or a
separate `secrets` field to distinguish secrets from plain vars at deploy time.

## Secret Rotation

### Platform secrets

Use the existing `scripts/update-secret.sh` pattern or call `docker secret`
directly:

```bash
# Rotate the DB password
printf "new-password" | docker secret create appa_pg_password_temp -
docker service update --secret-rm appa_pg_password \
  --secret-add source=appa_pg_password_temp,target=pg_password \
  appa_api
# ... wait for stabilization ...
docker secret rm appa_pg_password
printf "new-password" | docker secret create appa_pg_password -
docker service update --secret-rm appa_pg_password_temp \
  --secret-add source=appa_pg_password,target=pg_password \
  appa_api
docker secret rm appa_pg_password_temp
```

The `scripts/make-secrets.sh` script already implements this exact zero-downtime
rotation pattern and can be adapted for platform secrets.

### User app secrets

User app secrets are created per-deployment and cleaned up when the deployment
is removed. To rotate a user secret, the user sets a new value via
`project env set` and re-deploys. The old secret is cleaned up when the old
deployment is replaced.

## Scripts

### `scripts/make-secrets.sh`

Creates or updates Docker Swarm secrets from an `.env` file. Supports `--create`
(create new, skip existing) and `--update` (zero-downtime rotation via temp
secret pattern) modes, plus an optional `--prefix` for environment scoping.

```bash
# Create platform secrets from a staging env file
./scripts/make-secrets.sh .env.staging --prefix staging --create

# Rotate all secrets in production
./scripts/make-secrets.sh .env.production --prefix prod --update
```

### `scripts/update-secret.sh`

Rotates a single Docker secret with zero-downtime across a hardcoded service
list. Useful for ad-hoc rotation of individual platform secrets.

```bash
# Rotate the Cloudflare token
./scripts/update-secret.sh cloudflare_token "new-token-value" appa
```

For Phase 2, the service list should be dynamically discovered or replaced by
the pipeline's programmatic secret management.
