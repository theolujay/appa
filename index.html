One tool, two roles

The Operator and the Developer

Operator

Provision and manage servers from your terminal

Point appa at a fresh VPS. It handles SSH hardening, firewall rules, Docker, BuildKit, Caddy, and PostgreSQL. Everything provisioned with Ansible and Molecule-tested in CI.

$ appa server init demo --host 203.0.113.10 -i ~/.ssh/id_file
✓ Server "demo" created
✓ SSH target @203.0.113.10 set for "demo"
$ appa server preflight demo # checks SSH, RAM, OS
✓ All critical checks passed
$ appa server setup demo # hardens + provisions
Setup Complete
$ appa server status demo
✓ appa Stack running
$ appa server upgrade demo # pull & redeploy
Developer

Deploy apps without a Dockerfile or config file

Push a Git URL, a ZIP, or your local directory. appa detects your runtime, builds an optimized image, starts the container, and registers the subdomain. All automatic.

$ appa project init ./my-app --target demo
Project "my-app" created
$ appa deploy my-app
[build] Node.js 20 detected via Railpack
[build] Building image...
✓ Container started
✓ Route registered
Live at https://my-app.yourdomain.com

$ appa project logs my-app # stream runtime logs
$ appa project restart my-app

One person can wear both hats. The CLI handles both, even for an entire fleet of VPS instances.

Why appa

Terminal-first, dashboard-included

CLI-native fleet management

Manage your fleet with appa server setup, appa server apply, and appa server status. One binary handles any number of VPSs.

Ansible-backed security hardening

appa server setup runs Ansible playbooks for SSH hardening, UFW, auditd, kernel tuning, and fail2ban. Molecule-tested in CI, not a bash script hoping for the best.

Zero-config builds via Railpack

Railpack inspects your code and generates an optimized build plan. No Dockerfiles, no config. Ships source via rsync so you don't need a registry.

Browser-based dashboard

Monitor server status, deployment history, and live logs from any browser. Mirrors the CLI so there's no context switch.
